General

What is Simplex?

Simplex is a specification for describing work that autonomous agents will perform. It captures what needs to be done and how to know when it's done, without prescribing how to do it. It's designed to be interpreted by large language models directly, without formal parsing.

Who is Simplex for?

Simplex is for anyone building autonomous agent workflows, whether for software development, document processing, or other domains where agents work for extended periods without human intervention. It's particularly useful when you need specifications that are complete enough to act on without clarification.

Is Simplex a programming language?

No. Simplex has no formal grammar, no parser, and no compilation step. It's a semi-structured text format that LLMs interpret directly using landmarks (structural markers like FUNCTION, RULES, EXAMPLES). This is intentional—a formal grammar would contradict the principle of syntactic tolerance.

What version is current?

Version 0.5 is the current version. It introduces the DETERMINISM landmark for output variance control and consolidates the pillars from six to five. See the Version History for details on all versions.

Design Philosophy

Why "syntactic tolerance, semantic precision"?

LLMs are excellent at understanding intent despite typos, formatting inconsistencies, and notational variations. What they struggle with is ambiguity in meaning. Simplex leans into this: it forgives sloppy notation but requires precise meaning. If an agent would have to guess your intent, the specification is invalid.

Why are EXAMPLES mandatory?

Examples serve multiple purposes: they disambiguate rules, provide test cases, and ground the specification in concrete behavior. If the rules can be interpreted multiple ways, examples resolve the ambiguity. Without them, agents would have to guess what correct output looks like.

Why is there no composition construct?

This is an intentional research hypothesis. Simplex is designed for autonomous workflows where agents operate over extended periods. The hypothesis is that agents can infer task dependencies from context, potentially discovering structures the author didn't anticipate. Prescribed composition would constrain this emergent behavior.

If you need to suggest relationships between functions, use READS/WRITES for data dependencies, TRIGGERS for activation conditions, or HANDOFF for stage transitions.

What's the deal with DETERMINISM?

Added in v0.5, DETERMINISM addresses output reproducibility. When the same specification runs multiple times, should outputs be identical (strict), structurally equivalent (structural), or semantically equivalent (semantic)? This matters for testing, debugging, and pipeline reliability.

Why complexity limits?

Enforced simplicity is a core pillar. Specifications that exceed complexity limits (15 rules, 6 inputs, 200 chars per rule) must be decomposed. This is not a limitation, though. It's a forcing function and guardrail against ambiguity. Complexity that cannot be decomposed is complexity that is not yet understood.

Usage

How do I write a Simplex specification?

Start with a FUNCTION block naming your operation. Add RULES describing behavior as outcomes (not steps). Add DONE_WHEN with observable completion criteria. Add EXAMPLES covering every conditional branch in your rules. Add ERRORS specifying failure handling. That's a minimal valid spec.

What landmarks are required?

Five are always required: FUNCTION, RULES, DONE_WHEN, EXAMPLES, and ERRORS. EVAL is conditionally required when BASELINE is present. Everything else is optional.

When should I use DATA blocks?

Use DATA when the shape of inputs or outputs would otherwise be unclear. If your function takes a string and returns a string, you probably don't need DATA. If it takes a PolicyRule with specific fields and constraints, define it.

What's the difference between BASELINE and greenfield?

A greenfield spec describes new functionality. An evolutionary spec (with BASELINE) describes changes to existing functionality. BASELINE declares what exists now, what must be preserved, and what is being evolved. Use it when modifying existing systems.

How do I handle multi-agent coordination?

Use READS and WRITES to declare shared state dependencies. Use TRIGGERS to specify when an agent should pick up work. Use HANDOFF to describe what the next stage receives. Agents coordinate through shared memory without central orchestration.

Validation & Tooling

Is there a linter?

Yes. The Simplex linter validates structural requirements (required landmarks present), complexity limits (rule counts, input counts), and semantic criteria (example coverage). It produces errors, warnings, and suggestions.

What does the linter check?

  • Structural: At least one FUNCTION, required landmarks present, EVAL when BASELINE exists
  • Complexity: Rules count, input count, rule length, function count
  • Semantic: Example coverage for rule branches, DETERMINISM level validity
  • Evolution: BASELINE and EVAL field requirements when present

Can I use Simplex with my existing agent framework?

Yes. Simplex is a specification format, not an agent framework. Your agents read Simplex specs as input and interpret them. The spec is agnostic to how agents are implemented, what language they use, or what orchestration approach you take.

Comparison to Alternatives

How is Simplex different from prompting?

Prompts are conversational and often assume back-and-forth clarification. Simplex specs are complete—they contain everything an agent needs to act without additional context. This is essential for autonomous workflows where agents work for extended periods.

How is Simplex different from programming languages?

Programming languages describe how to do something. Simplex describes what to do and how to verify it's done. This leaves implementation choices to agents, who may discover better approaches than the spec author anticipated.

How is Simplex different from formal specification languages?

Formal specifications (TLA+, Z, Alloy) are mathematically precise but require specialized knowledge to write and verify. Simplex is designed for LLM interpretation with syntactic tolerance—it's accessible to anyone who can describe behavior in semi-structured prose.

Should I use Simplex or just write better prompts?

For one-off tasks with human supervision, prompts are fine. For autonomous workflows where agents work independently for extended periods, Simplex provides the structure and completeness that prompts lack. The decision depends on your use case.
